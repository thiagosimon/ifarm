# ADR-006: Pagar.me v5 as Payment Gateway

## Status: Accepted

## Context

iFarm operates as a marketplace connecting farmers (sellers) to buyers. Every transaction involves splitting the payment between the farmer, the platform (commission), and potentially logistics partners. Brazilian payment methods (PIX, boleto bancario, credit card installments) must be supported natively. The payment gateway must handle marketplace-style splits where the platform takes a commission on the total invoice value including taxes (RFN-001), with a 7-business-day holdback period (RFN-002). The split must be configured before the charge is created, not after (RFN-003).

## Decision

Use Pagar.me v5 as the exclusive payment gateway for all marketplace transactions.

Integration details:
- **Split rules**: Configured at charge creation time using Pagar.me's native marketplace split feature. Platform commission is calculated on the total NF value (including taxes) per RFN-001.
- **Payment methods**: PIX (instant, 15-minute expiry), boleto bancario (3-day expiry), credit card (up to 12x installments)
- **Webhooks**: All payment status changes are received via Pagar.me webhooks with HMAC-SHA256 signature verification. Each webhook is idempotent using `pagarmeTransactionId` as a unique constraint (RFN-004).
- **Holdback**: 7 business days (excluding Brazilian holidays) before funds are released to sellers (RFN-002). Holiday calendar is maintained in the commission service.
- **PCI compliance**: iFarm never stores PAN or CVV (RFN-005). Card tokenization is handled entirely by Pagar.me's client-side SDK.
- **Recipient management**: Each farmer is registered as a Pagar.me recipient with their bank account details for automatic disbursement.

## Alternatives Rejected

- **Stripe**: Rejected because Stripe's Brazilian operations are newer and lack native marketplace split functionality equivalent to Pagar.me's. Stripe Connect exists but requires additional configuration for Brazilian tax compliance and does not natively support boleto bancario with split.
- **Mercado Pago**: Rejected due to less flexible split rules, mandatory Mercado Pago branding requirements, and slower settlement times compared to Pagar.me.
- **Asaas**: Rejected because Asaas's marketplace features are less mature than Pagar.me v5, and their API documentation is significantly less comprehensive.
- **Building direct bank integrations**: Rejected due to the enormous complexity of integrating directly with Brazilian banks for PIX, boleto generation, and credit card processing. PCI DSS compliance alone would be a multi-month, multi-hundred-thousand-dollar effort.
- **iugu**: Considered as a backup option. Rejected because Pagar.me v5's split marketplace feature is more mature and better documented.

## Consequences

- Native marketplace split eliminates the need for iFarm to handle money movement between parties.
- Webhook-driven architecture means payment status updates are asynchronous; the system must handle delayed or missing webhooks gracefully (see docs/runbooks/payment-stuck.md).
- Pagar.me is a single point of failure for all payments. If Pagar.me is down, no transactions can be processed. Mitigation: monitor status.pagar.me and implement manual reconciliation procedures.
- The platform commission percentage and holdback period are configurable per product category to support future pricing flexibility.
- All Pagar.me API calls are logged with correlation IDs for debugging and reconciliation.
