# ADR-005: Keycloak as Identity Provider

## Status: Accepted

## Context

iFarm needs a centralized identity and access management solution supporting multiple user roles (farmer, buyer, admin, logistics partner, agronomist), multi-tenant organization management, OAuth 2.0/OIDC flows for web and mobile clients, and social login capabilities. The platform serves Brazilian agribusiness users who may register via email, phone, or CPF. Cost predictability is critical for a startup -- per-user pricing from SaaS identity providers becomes prohibitive as the user base grows into tens of thousands of farmers.

## Decision

Use Keycloak 24 as the self-hosted identity provider for all authentication and authorization.

Configuration:
- **Realm**: Single `ifarm` realm containing all user types
- **Clients**: Separate OIDC clients for each frontend (Flutter mobile, web admin, farmer portal, buyer portal, logistics portal)
- **Roles**: Realm-level roles (`farmer`, `buyer`, `admin`, `logistics`, `agronomist`) with composite roles for fine-grained permissions
- **Identity brokering**: Social login via Google and Facebook; CPF-based login via custom authenticator SPI
- **Token configuration**: Access tokens (JWT) with 15-minute expiry; refresh tokens with 30-day expiry; offline tokens for mobile
- **User federation**: Custom user storage SPI connecting to iFarm's user-service for profile data synchronization
- **Deployment**: Keycloak runs as a Kubernetes Deployment with PostgreSQL as its backing database (separate from the application PostgreSQL)

## Alternatives Rejected

- **Auth0**: Rejected primarily due to cost. Auth0's pricing is per-monthly-active-user; at 50,000+ farmers, the cost would exceed $10,000/month. Keycloak is open-source with zero per-user cost. Auth0's developer experience is superior, but the cost difference is too significant for a startup.
- **Firebase Authentication**: Rejected due to limited customization for Brazilian-specific auth flows (CPF-based login, Pagar.me KYC integration), vendor lock-in to Google Cloud, and no self-hosted option for data sovereignty (LGPD compliance).
- **Supabase Auth**: Rejected because it lacks the enterprise IAM features needed: fine-grained RBAC, identity brokering, custom authenticator SPIs, and organization management.
- **Custom-built auth**: Rejected because building a secure, standards-compliant OAuth 2.0/OIDC server is a multi-month effort fraught with security risks. Keycloak has been battle-tested by thousands of organizations.
- **Ory Kratos/Hydra**: Considered seriously. Rejected because the split architecture (Kratos for identity, Hydra for OAuth) adds operational complexity, and the Node.js/NestJS ecosystem has more mature Keycloak integration libraries.

## Consequences

- Zero per-user cost regardless of user base growth, making cost predictable.
- The team must maintain and operate Keycloak (upgrades, backup, monitoring), adding operational burden.
- Keycloak's admin UI provides user management, role assignment, and session management out of the box.
- Custom authenticator SPIs (CPF login) require Java development, which is outside the team's primary skillset.
- All services validate JWT tokens issued by Keycloak using a shared NestJS guard (`@nestjs/passport` + `passport-jwt`).
- LGPD compliance is simplified because all identity data stays within infrastructure the team controls.
