# ADR-011: English Field Naming

## Status: Accepted

## Context

iFarm is a Brazilian agribusiness platform, and the natural inclination is to use Portuguese field names reflecting Brazilian concepts (e.g., `cpf`, `cnpj`, `nota_fiscal`, `boleto`). However, the platform's long-term vision includes expansion to other Latin American markets (Argentina, Paraguay, Colombia) where Portuguese is not spoken and where equivalent concepts have different local names. Additionally, the development team works with international libraries, tools, and AI coding assistants that operate primarily in English. Mixed-language codebases (Portuguese field names in TypeScript code with English keywords and library APIs) create cognitive friction and make the codebase harder to navigate.

## Decision

All field names, variable names, function names, API parameters, database columns, and event names are written in English. Brazilian-specific concepts are mapped to universal English equivalents.

Mapping examples:
| Brazilian concept | English field name | Notes |
|---|---|---|
| CPF | `federalTaxId` | Individual tax identification |
| CNPJ | `businessRegistrationId` | Business tax identification |
| Nota Fiscal | `invoice` | Tax invoice |
| Boleto | `bankSlip` or `boleto` (payment method enum) | Payment method preserved as enum value |
| NCM | `tariffCode` | Harmonized System classification |
| Inscrição Estadual | `stateRegistrationId` | State-level tax registration |
| Razão Social | `legalName` | Official registered business name |
| Nome Fantasia | `tradeName` | Business trading name |
| Endereço | `address` | Standard address object |
| Safra | `harvestSeason` | Agricultural season/crop cycle |
| Talhão | `fieldPlot` | Specific area within a farm |
| Fazenda | `farm` | Agricultural property |

Exceptions where Portuguese terms are preserved:
- Payment method enum values: `PIX`, `BOLETO`, `CREDIT_CARD` (PIX is a proper noun, BOLETO is a recognized payment method)
- Tax type enum values: `ICMS`, `PIS`, `COFINS`, `IPI` (Brazilian tax acronyms are universally recognized in their Portuguese form)

## Alternatives Rejected

- **Portuguese field names everywhere**: Rejected because it limits LATAM expansion, creates mixed-language code (Portuguese fields in English-syntax TypeScript), and reduces effectiveness of AI coding assistants that are trained primarily on English codebases.
- **Bilingual approach (Portuguese in DB, English in API)**: Rejected because maintaining a translation layer between database and API adds complexity, introduces mapping bugs, and doubles the field name vocabulary developers must memorize.
- **Portuguese field names with English comments**: Rejected because comments go stale; field names are the primary documentation of their purpose.

## Consequences

- The codebase is consistently English, reducing cognitive friction and improving readability for international collaborators and AI tools.
- LATAM expansion requires only adding new enum values (e.g., `CUIT` for Argentina's tax ID) rather than renaming existing fields.
- Brazilian developers must learn the English mappings, which has a short learning curve. The mapping table above serves as a reference.
- `federalTaxId` is used for both CPF (11 digits, individuals) and CNPJ (14 digits, businesses); the `userType` field (`INDIVIDUAL` vs `BUSINESS`) disambiguates.
- API documentation must clearly explain the Brazilian equivalents for non-obvious field names (e.g., `tariffCode` = NCM).
- Per rule RSD-001, `federalTaxId` and `businessRegistrationId` are encrypted at rest using AES-256-GCM.
