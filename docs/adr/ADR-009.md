# ADR-009: No Service Mesh in Phase 1

## Status: Accepted

## Context

Service meshes (Istio, Linkerd) provide features like mutual TLS, traffic management, observability, and circuit breaking for microservice architectures. iFarm runs 17 microservices on Kubernetes, and the team evaluated whether a service mesh would provide meaningful benefits at the current scale. The evaluation considered operational complexity, resource overhead, debugging difficulty, and the team's Kubernetes expertise.

## Decision

Do not deploy a service mesh in Phase 1 of the iFarm platform. Revisit this decision when the service count exceeds 25 or when specific service mesh features become necessary.

Current alternatives for service mesh features:
- **mTLS**: Not required in Phase 1; services communicate within a private Kubernetes network namespace. Network policies restrict inter-namespace traffic.
- **Traffic management**: Kubernetes Services and Ingress provide sufficient traffic routing. Canary deployments are handled via Kubernetes rollout strategies (rolling update with maxSurge/maxUnavailable).
- **Observability**: Prometheus + Grafana for metrics, Loki for logs, and OpenTelemetry for distributed tracing provide adequate observability without a service mesh sidecar.
- **Circuit breaking**: Implemented at the application layer using NestJS interceptors and axios retry logic, providing sufficient resilience for current traffic volumes.
- **Rate limiting**: Handled by Redis-based rate limiting in the API gateway, not requiring mesh-level controls.

## Alternatives Rejected

- **Istio**: Rejected due to significant resource overhead (Envoy sidecar per pod adds ~50MB memory and ~10ms latency per hop), complex configuration (CRDs, VirtualServices, DestinationRules), and steep debugging curve. With fewer than 20 services, the operational cost outweighs the benefits.
- **Linkerd**: Lighter than Istio but still adds sidecar overhead and operational complexity. Rejected for the same scale-based reasoning.
- **Consul Connect**: Rejected due to additional HashiCorp infrastructure requirements and the team's lack of Consul expertise.

## Consequences

- Reduced operational complexity: no sidecar injection, no mesh control plane to maintain, no mesh-specific debugging.
- Lower resource consumption: no Envoy sidecars consuming memory and CPU per pod.
- The team must implement circuit breaking, retry logic, and timeout handling at the application layer (NestJS interceptors/guards).
- No automatic mTLS between services; if mTLS becomes a compliance requirement, the team must either adopt a mesh or configure certificate-based auth manually.
- When the platform scales beyond 25 services or requires advanced traffic management (A/B testing, fault injection), a service mesh should be reconsidered.
- Distributed tracing relies on application-level OpenTelemetry instrumentation rather than automatic sidecar-based tracing.
