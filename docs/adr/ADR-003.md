# ADR-003: RabbitMQ as Event Bus

## Status: Accepted

## Context

iFarm's microservice architecture requires asynchronous communication between services for event-driven workflows. Key flows include: quotation accepted triggers order creation, payment confirmed triggers commission calculation, order shipped triggers notification dispatch. The event bus must support topic-based routing, dead-letter queues for failed message handling, and be operationally manageable by a small team. The platform currently runs fewer than 20 services and processes moderate message volumes (estimated thousands per day in the first year, not millions).

## Decision

Use RabbitMQ 3.13 as the primary event bus for all inter-service asynchronous communication.

Configuration details:
- **Exchange type**: Topic exchanges for flexible routing (e.g., `order.created`, `payment.confirmed`, `quotation.accepted`)
- **Dead-letter queues (DLQ)**: Every queue has a corresponding `dlq.{queue-name}` for failed messages after 3 retry attempts
- **Message persistence**: All messages are published as persistent (delivery mode 2) to survive broker restarts
- **Consumer acknowledgment**: Manual ack after successful processing; nack with requeue=false on permanent failure (routes to DLQ)
- **Deployment**: Single RabbitMQ instance in development; clustered (3-node) in production via Kubernetes StatefulSet

## Alternatives Rejected

- **Apache Kafka**: Rejected for the current scale. Kafka excels at high-throughput stream processing (millions of events/second) but introduces significant operational overhead: ZooKeeper/KRaft management, partition rebalancing, consumer group coordination. iFarm's message volume does not justify this complexity. Kafka may be reconsidered when IoT telemetry ingestion scales beyond RabbitMQ's comfortable throughput.
- **Amazon SQS/SNS**: Rejected to avoid hard AWS vendor lock-in and to keep infrastructure portable across cloud providers. SQS also lacks native topic-based routing -- SNS+SQS fan-out adds architectural complexity that RabbitMQ handles natively.
- **Redis Streams**: Rejected because Redis is already used for caching, rate limiting, and BullMQ job queues (see ADR-012). Overloading Redis with event bus responsibilities would create a single point of failure for too many concerns. Additionally, Redis Streams lack mature dead-letter queue support.
- **NATS**: Rejected due to smaller community adoption in the Node.js/NestJS ecosystem and less mature management tooling compared to RabbitMQ.

## Consequences

- Services communicate asynchronously via topic exchanges, enabling loose coupling and independent deployability.
- Dead-letter queues provide a safety net for failed messages; a DLQ investigation runbook is required (see docs/runbooks/dlq-investigation.md).
- RabbitMQ Management UI (port 15672) provides visibility into queue depths, consumer counts, and message rates.
- The NestJS microservices transport layer has first-class RabbitMQ support, reducing boilerplate.
- The team must monitor queue depth and consumer lag via Prometheus metrics exported by the RabbitMQ Prometheus plugin.
- If message volumes grow significantly (IoT scale-out), the team should revisit Kafka as a complementary system for high-throughput streams while keeping RabbitMQ for command/event patterns.
