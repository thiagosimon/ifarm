# ADR-010: Static Tax Table in MVP

## Status: Accepted

## Context

iFarm's marketplace involves agricultural products subject to Brazilian taxation (ICMS, PIS, COFINS, IPI, and others). Accurate tax calculation is essential for invoice generation (NF-e) and commission calculations (RFN-001: commission on total NF value including taxes). Full tax automation services like Taxcel, Avalara, or TaxJar cost between R$2,000-10,000/month and require integration effort. During the MVP phase, iFarm operates with a limited product catalog (seeds, fertilizers, agrochemicals, machinery) with well-known tax classifications (NCM codes). The number of unique tax scenarios is manageable without a dynamic tax engine.

## Decision

Use a static tax table for the MVP phase, with plans to integrate Taxcel in Phase 1.5 (F1.5) when product catalog diversity increases.

Implementation:
- **Tax table location**: A configuration file or database collection mapping NCM codes (tariffCode, per RCQ-001) to tax rates by state (origin/destination ICMS), product category, and tax regime (Simples Nacional, Lucro Presumido, Lucro Real)
- **Scope**: ICMS (interstate and intrastate), PIS, COFINS for the initial product categories
- **Maintenance**: Tax rates are updated manually by the operations team when legislation changes; updates are versioned and auditable
- **Validation**: Every quotation line item requires a tariffCode (NCM) per RCQ-001; the tax calculation service looks up rates from the static table
- **Fallback**: If an NCM code is not found in the static table, the quotation is flagged for manual review

## Alternatives Rejected

- **Taxcel integration from day one**: Rejected due to monthly licensing costs (R$3,000+/month minimum) that are not justified during the MVP phase with a limited product catalog. Taxcel will be integrated in F1.5 when the catalog expands beyond the initial categories.
- **Avalara AvaTax**: Rejected due to higher cost than Taxcel for Brazilian tax scenarios and less mature Brazilian tax compliance compared to local providers.
- **No tax calculation (leave to accountant)**: Rejected because RFN-001 requires commission calculation on total NF value including taxes, which means the platform must know the tax amounts at quotation/order time.
- **Real-time SEFAZ consultation**: Rejected because SEFAZ APIs are not designed for real-time tax rate lookups; they are for NF-e authorization, not tax calculation.

## Consequences

- Zero additional cost for tax calculation during the MVP phase.
- The operations team must monitor Brazilian tax legislation changes and update the static table accordingly. This is manageable for a limited product catalog but will not scale.
- Tax calculations may have minor inaccuracies for edge cases (tax substitution, special regimes) that a dynamic engine would handle. Accepted risk for MVP.
- The tariffCode (NCM) requirement on every line item (RCQ-001) ensures data readiness for Taxcel integration in F1.5.
- When Taxcel is integrated, the static table becomes the fallback for cases where Taxcel is unavailable or returns errors.
- Integration effort for Taxcel in F1.5 is estimated at 2-3 weeks, primarily API integration and response mapping.
