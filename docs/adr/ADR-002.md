# ADR-002: PostgreSQL for Financial Domains

## Status: Accepted

## Context

iFarm's marketplace involves real money transactions: order payments (PIX, boleto, credit card via Pagar.me), commission calculations, split disbursements, holdback periods, and financial reconciliation. These operations require strict ACID guarantees -- a partial commission calculation or a lost payment record is unacceptable. Brazilian regulatory requirements (fiscal compliance, audit trails) further demand transactional integrity and relational consistency. The platform must also maintain accurate ledger-style records for holdback periods (7 business days excluding Brazilian holidays, per RFN-002) and Pagar.me split configurations.

## Decision

Use PostgreSQL 16 as the datastore for all financial domains, including:

- **Payment Service** -- payment transactions, Pagar.me webhook records, idempotency keys
- **Order Service** -- orders, order line items, order state machine transitions
- **Commission Service** -- commission calculations, split configurations, holdback schedules
- **Financial Reconciliation** -- ledger entries, Pagar.me settlement reconciliation

All IDs in PostgreSQL domains use UUID v4 to avoid sequential ID enumeration attacks on financial endpoints.

Soft delete is mandatory for all financial and KYC-related records to preserve audit trails.

## Alternatives Rejected

- **MongoDB for everything**: Rejected because MongoDB's eventual consistency model and lack of multi-document ACID transactions (prior to 4.0, and still with performance caveats in 4.0+) create unacceptable risk for financial operations. A race condition in commission calculation could result in incorrect splits sent to Pagar.me.
- **CockroachDB**: Rejected due to operational complexity, smaller ecosystem of tools and ORMs in the Node.js/TypeScript world, and higher hosting costs for the team's current scale.
- **MySQL**: Rejected in favor of PostgreSQL's superior support for UUID types, JSONB for hybrid needs, and richer extension ecosystem.

## Consequences

- Financial data benefits from full ACID transactions, foreign key constraints, and relational integrity.
- The team uses TypeORM or Prisma (TBD per service) for migrations and type-safe queries.
- Cross-domain queries between PostgreSQL (financial) and MongoDB (non-financial) require event-driven synchronization via RabbitMQ (see ADR-003) rather than direct joins.
- Soft delete adds a `deletedAt` column to financial tables; all queries must filter on `deletedAt IS NULL` by default.
- UUID v4 primary keys increase storage slightly compared to auto-increment integers but eliminate enumeration risks on public-facing APIs.
