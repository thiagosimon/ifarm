# ADR-012: BullMQ for Async Jobs

## Status: Accepted

## Context

iFarm requires background job processing for various asynchronous tasks that do not fit the event-driven RabbitMQ pattern (ADR-003). These include scheduled jobs (daily commission calculations, holdback release checks, expired quotation cleanup), delayed jobs (send reminder 2 hours before PIX expiry, retry failed webhook delivery), and rate-limited jobs (bulk email sending via AWS SES, bulk SMS via WhatsApp Cloud API). These jobs need persistence (survive service restarts), retry logic with exponential backoff, concurrency control, and job progress tracking. The job queue system must integrate naturally with the NestJS/TypeScript ecosystem.

## Decision

Use BullMQ as the job queue system for all asynchronous background jobs across iFarm services.

Implementation details:
- **Backing store**: Redis 7 (the same Redis cluster used for caching and rate limiting, but using a dedicated database number or key prefix `bull:` to isolate job data)
- **NestJS integration**: `@nestjs/bullmq` module for declarative queue/worker registration via decorators
- **Job types**:
  - **Immediate jobs**: Fire-and-forget processing (e.g., send notification after order confirmation)
  - **Delayed jobs**: Execute after a specified delay (e.g., cancel unpaid PIX order after 15 minutes)
  - **Scheduled/repeatable jobs**: Cron-based recurring jobs (e.g., daily holdback release check at 06:00 UTC)
  - **Rate-limited jobs**: Controlled throughput (e.g., max 10 emails/second to respect SES limits)
- **Retry strategy**: Failed jobs retry up to 3 times with exponential backoff (1s, 4s, 16s). After max retries, jobs move to the failed set for manual inspection.
- **Monitoring**: Bull Board UI for job inspection, mounted at `/admin/queues` behind admin authentication

## Alternatives Rejected

- **RabbitMQ delayed messages**: Rejected because RabbitMQ's delayed message plugin is a community plugin (not core), has limitations on delay precision, and conflates the event bus concern with the job queue concern. Keeping event-driven communication (RabbitMQ) separate from job scheduling (BullMQ) provides clearer architectural boundaries.
- **Agenda.js (MongoDB-backed)**: Rejected because Agenda.js stores jobs in MongoDB, which would couple job scheduling to the document database. BullMQ's Redis backing provides significantly lower latency for job claiming and completion tracking. Agenda.js also has a less active maintenance community.
- **node-cron / cron jobs**: Rejected because simple cron-based scheduling lacks persistence (jobs lost on restart), retry logic, concurrency control, and distributed locking. In a Kubernetes environment with multiple pod replicas, a cron job would execute on every pod simultaneously without external coordination.
- **AWS SQS + CloudWatch Events**: Rejected to avoid AWS vendor lock-in and to keep the job infrastructure consistent with the self-hosted approach (Redis is already part of the stack).
- **Temporal.io**: Considered for its powerful workflow orchestration capabilities. Rejected because Temporal's operational overhead (requires its own server cluster) is too heavy for iFarm's current job scheduling needs. Temporal may be reconsidered if complex multi-step workflows (e.g., multi-party escrow with conditional release) are needed in the future.

## Consequences

- BullMQ provides a TypeScript-first API that integrates seamlessly with NestJS, reducing boilerplate for queue/worker definitions.
- Redis becomes a more critical piece of infrastructure, as it now stores both cache data and job queues. Redis persistence (AOF or RDB) must be configured to prevent job loss on Redis restart.
- Job processing is distributed across service replicas automatically; BullMQ uses Redis-based locking to ensure exactly-once job claiming.
- Bull Board provides operational visibility into queue depths, processing rates, and failed jobs without building custom dashboards.
- Redis memory consumption increases with job volume; the team must configure job retention policies (e.g., remove completed jobs after 24 hours, keep failed jobs for 7 days).
- BullMQ does not provide cross-queue workflow orchestration (job A triggers job B triggers job C); complex workflows must be orchestrated at the application layer or via RabbitMQ events.
