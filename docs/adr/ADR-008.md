# ADR-008: Monorepo with Turborepo

## Status: Accepted

## Context

iFarm consists of 17 NestJS microservices, 4 Next.js web applications, 1 Flutter mobile app, and multiple shared packages (DTOs, utilities, configurations). Managing these as separate repositories would create significant overhead: duplicated CI/CD configurations, version drift on shared types, difficult cross-service refactoring, and cumbersome dependency updates. The team needs a monorepo strategy that enables shared TypeScript types between services and frontends, incremental builds for fast CI, and a unified development experience.

## Decision

Use a monorepo managed by Turborepo with npm workspaces.

Repository structure:
```
/
  apps/           # Next.js web applications and Flutter mobile
  services/       # NestJS microservices
  packages/       # Shared packages (DTOs, common utilities, configs)
  infra/          # Kubernetes manifests, Terraform, Docker configs
  docs/           # Documentation, ADRs, runbooks
  turbo.json      # Turborepo pipeline configuration
  package.json    # Root workspace configuration
```

Key conventions:
- **Shared types**: `packages/dto` contains TypeScript interfaces and DTOs shared between services and web frontends
- **Common utilities**: `packages/common` contains shared guards, interceptors, decorators, and utilities
- **Build pipeline**: Turborepo orchestrates `build`, `test`, `lint` tasks with dependency-aware caching
- **CI caching**: Turborepo's remote cache (via Vercel or self-hosted) stores build artifacts across CI runs, reducing build times by 60-80% for unchanged packages
- **Dependency management**: npm workspaces handle internal package resolution; `package.json` in each workspace declares its dependencies

## Alternatives Rejected

- **Nx**: Considered seriously as the primary alternative. Rejected because Nx's configuration overhead is higher, its plugin system is more opinionated, and Turborepo's simpler mental model (just a task runner with caching) is better suited to the team's needs. Nx's built-in code generators are powerful but add complexity the team does not need in the MVP phase.
- **Lerna**: Rejected because Lerna's maintenance has been inconsistent (it was deprecated, then revived by Nx). Turborepo provides the same monorepo orchestration with better caching and a more active development roadmap.
- **Polyrepo (separate repositories)**: Rejected due to the overhead described in the Context section. Shared TypeScript types between 17 services and 4 web apps would require publishing to a private npm registry, adding latency and version management complexity.
- **Bazel**: Rejected due to steep learning curve and over-engineering for a team of this size. Bazel excels at massive monorepos (thousands of packages); iFarm's ~25 packages do not warrant this complexity.
- **pnpm workspaces + custom scripts**: Considered for its faster install times. Rejected because Turborepo's task caching and pipeline orchestration provide significant value beyond what custom scripts can achieve, and npm workspaces are sufficient for iFarm's dependency resolution needs.

## Consequences

- Developers can make cross-cutting changes (e.g., rename a DTO field) in a single commit affecting all services and frontends simultaneously.
- Turborepo's caching reduces CI build times significantly; only changed packages and their dependents are rebuilt.
- All services share the same TypeScript version, ESLint config, and Prettier config, ensuring consistency.
- The monorepo grows in size over time; git clone and initial install times increase. Mitigated by shallow clones in CI and Turborepo's pruning feature for Docker builds.
- Flutter (Dart) does not participate in the npm workspace graph; it lives in `apps/` but is built independently. API contract synchronization between Dart and TypeScript requires discipline.
- New developers must understand the monorepo structure and Turborepo commands (`npx turbo run build --filter=payment-service`) before being productive.
