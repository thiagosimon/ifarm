# ADR-004: NestJS for All Microservices

## Status: Accepted

## Context

iFarm requires a consistent backend framework across 17 microservices to ensure developer productivity, code reuse, and maintainability. The team is TypeScript-first and needs a framework that provides dependency injection, modular architecture, built-in support for microservice transports (RabbitMQ), and a rich ecosystem of community modules. Consistency across services reduces context-switching costs when developers move between services and enables shared libraries for cross-cutting concerns (logging, health checks, authentication guards).

## Decision

Use NestJS 10 as the framework for all backend microservices without exception.

Key conventions:
- **Dependency Injection**: All services, repositories, and external integrations are injected via NestJS DI container
- **Module structure**: Each service follows `{domain}.module.ts` -> `{domain}.controller.ts` -> `{domain}.service.ts` -> `{domain}.repository.ts`
- **Microservice transport**: `@nestjs/microservices` with RabbitMQ transport for event consumers
- **Validation**: `class-validator` + `class-transformer` for DTO validation on all inbound payloads
- **Configuration**: `@nestjs/config` with `.env` files and Joi schema validation at startup
- **Health checks**: `@nestjs/terminus` with readiness and liveness probes for Kubernetes
- **Shared packages**: Common DTOs, decorators, guards, and interceptors live in `packages/` within the monorepo (see ADR-008)

## Alternatives Rejected

- **Express.js (raw)**: Rejected because raw Express provides no opinions on architecture, leading to inconsistent patterns across 17 services. Every team member would need to agree on DI, validation, module boundaries, and testing conventions from scratch.
- **Fastify (raw)**: Same architectural concerns as Express. While Fastify is faster in benchmarks, the performance difference is negligible for iFarm's workload, and NestJS can use Fastify as its underlying HTTP adapter if needed later.
- **tRPC**: Rejected because iFarm's mobile app (Flutter) and external integrations (Pagar.me webhooks) require standard REST/HTTP endpoints. tRPC's TypeScript-to-TypeScript approach does not serve non-TypeScript clients.
- **Go/Rust microservices**: Rejected due to the team's TypeScript expertise. Introducing a second language would slow development, complicate hiring, and eliminate the benefit of shared TypeScript types between frontend and backend.
- **Spring Boot (Java/Kotlin)**: Rejected due to slower development velocity for a startup, heavier resource consumption per service, and no shared type system with the TypeScript frontends.

## Consequences

- All 17 services share identical project structure, reducing onboarding time for new developers.
- Shared packages (`packages/common`, `packages/dto`) can be imported by any service with zero duplication.
- NestJS's built-in testing utilities (`@nestjs/testing`) provide consistent testing patterns across services.
- The team is locked into the NestJS ecosystem; if NestJS development stalls, migration would be costly.
- NestJS's decorator-heavy approach can obscure control flow for developers unfamiliar with the framework.
- TypeScript compilation adds a build step, but Turborepo's caching (see ADR-008) mitigates rebuild times.
