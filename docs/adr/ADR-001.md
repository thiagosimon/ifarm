# ADR-001: MongoDB for Non-Financial Domains

## Status: Accepted

## Context

iFarm operates across multiple domains with varying data shape requirements. Quotation workflows involve deeply nested structures (line items, specifications, unit conversions, attachments) that evolve rapidly during early product development. IoT telemetry data from farm sensors arrives in semi-structured payloads that differ by device manufacturer and firmware version. Trying to force these into rigid relational schemas would require frequent migrations, slow down iteration, and introduce unnecessary join complexity.

## Decision

Use MongoDB as the primary datastore for all non-financial domains, including:

- **Quotation Service** -- quotations with nested line items, product specs, negotiation history
- **Product Catalog** -- products with variable attributes per category (seeds vs. machinery vs. chemicals)
- **IoT/Telemetry Service** -- sensor readings with heterogeneous payloads
- **Notification Service** -- notification templates and delivery logs
- **Content/CMS** -- articles, tutorials, and media metadata

MongoDB Atlas is the managed deployment target, providing automated backups, point-in-time recovery, and auto-scaling.

Mongoose is used as the ODM across all NestJS services for schema validation, middleware hooks, and type safety via TypeScript interfaces.

## Alternatives Rejected

- **PostgreSQL with JSONB columns**: Considered for its ability to store semi-structured data alongside relational data. Rejected because it would create a split-brain situation where some columns are relational and others are document-style, complicating query patterns. The financial domains already use PostgreSQL (see ADR-002), so this would blur the boundary between the two storage strategies.
- **DynamoDB**: Rejected due to vendor lock-in to AWS, complex pricing model for unpredictable read/write patterns, and poor developer experience for ad-hoc queries during development.
- **Single database (PostgreSQL for everything)**: Rejected because the overhead of maintaining relational schemas for rapidly evolving document structures would slow iteration significantly in the MVP phase.

## Consequences

- Services in non-financial domains benefit from flexible schemas and fast iteration cycles.
- The team must enforce schema validation at the application layer (Mongoose schemas) since MongoDB does not enforce it by default.
- Cross-domain queries between MongoDB and PostgreSQL require application-level joins or event-driven denormalization.
- MongoDB ObjectId is the native ID format for these domains; PostgreSQL domains use UUID v4 (see ADR-002).
- Operational complexity increases slightly due to running two database engines, mitigated by using managed services (Atlas + managed PostgreSQL).
