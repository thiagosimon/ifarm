# ADR-013: CLAUDE.md per Project

## Status: Accepted

## Context

iFarm uses Claude Code (Anthropic's AI coding assistant) extensively for development across all 17 microservices, 4 web frontends, and the Flutter mobile app. Each time a developer starts a new Claude Code session, the AI agent has no memory of prior decisions, conventions, critical business rules, or architectural patterns. Developers repeatedly explain the same context: field naming conventions (ADR-011), critical financial rules (RFN-001 through RFN-005), database choices per domain (ADR-001, ADR-002), and monorepo structure (ADR-008). This repetition wastes developer time and risks inconsistent outputs when context is incompletely conveyed.

## Decision

Maintain a `CLAUDE.md` file at the root of the monorepo (and optionally within individual service/app directories) that serves as persistent context for Claude Code sessions.

The root `CLAUDE.md` contains:
- **Identity**: Platform name, tech stack summary, infrastructure components, and external integrations
- **Architecture**: Monorepo structure, microservice count, event-driven patterns, database strategy
- **Conventions**: Field naming rules, ID formats, API versioning, pagination, timestamp format, error response structure, soft delete policy
- **Critical rules**: Numbered business rules (RFN-xxx for financial, RCQ-xxx for quotation, RSD-xxx for security, RLGPD-xxx for LGPD compliance) that the AI must never violate
- **Commands**: Common development commands (build, test, local infra)

Optional per-service `CLAUDE.md` files can extend the root context with service-specific details:
- Domain model and entity relationships
- Service-specific environment variables
- External API integration details
- Testing commands and fixtures

## Alternatives Rejected

- **No context file (explain each session)**: Rejected because it wastes 5-10 minutes per session on context setting and risks incomplete context leading to code that violates conventions or business rules.
- **README.md as context**: Rejected because README.md serves a different audience (human developers reading documentation) and has different content priorities. Claude Code specifically looks for `CLAUDE.md` as its context file. Overloading README.md with AI-specific context would degrade its usefulness for humans.
- **Custom GPT / system prompt**: Rejected because the context must live in version control alongside the code it describes. A system prompt stored outside the repo would drift from the actual codebase and cannot be reviewed in pull requests.
- **Confluence / Notion documentation**: Rejected because Claude Code cannot access external documentation platforms during a coding session. The context must be in the repository filesystem.

## Consequences

- Every Claude Code session starts with full project context, ensuring consistent outputs aligned with architecture decisions and business rules.
- The `CLAUDE.md` file becomes part of the code review process; changes to conventions or rules require updating the file in the same PR.
- Developers must maintain `CLAUDE.md` as the project evolves; stale context is worse than no context because it leads to confident but incorrect AI outputs.
- The root `CLAUDE.md` must remain concise (under 100 lines) to avoid context window waste; detailed documentation belongs in `docs/` and is referenced by filename.
- Per-service `CLAUDE.md` files are optional and should only be created when a service has significant domain-specific context not captured in the root file.
- This approach is specific to Claude Code; other AI coding tools may not recognize `CLAUDE.md`. If the team adopts additional AI tools, equivalent context files may be needed.
