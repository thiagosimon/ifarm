# ADR-007: Flutter for Mobile

## Status: Accepted

## Context

iFarm's primary users are farmers operating in rural Brazil with intermittent or absent internet connectivity. The mobile app must function offline for core workflows (viewing quotations, checking product catalogs, reviewing order history) and synchronize when connectivity returns. The app must run on both Android and iOS with a native-quality experience. The development team is small and cannot maintain two separate native codebases. Android is the dominant platform (estimated 85%+ of target users), but iOS support is required for larger agribusiness buyers.

## Decision

Use Flutter 3.x for the iFarm mobile application with an offline-first architecture.

Implementation details:
- **Offline storage**: Hive or Isar for local structured data persistence, enabling offline access to quotations, products, and order history
- **Sync strategy**: Queue-based synchronization -- mutations made offline are queued locally and replayed when connectivity returns, with conflict resolution favoring server state for financial data and last-write-wins for non-critical data
- **State management**: Riverpod for dependency injection and state management
- **API layer**: Dio HTTP client with interceptors for JWT token refresh (tokens from Keycloak), request retry, and offline queue management
- **Push notifications**: OneSignal SDK for cross-platform push notifications
- **Platform targets**: Android (minSdk 21) and iOS (14.0+)
- **Build and distribution**: Fastlane for automated builds, Firebase App Distribution for beta testing, Google Play Store and Apple App Store for production

## Alternatives Rejected

- **React Native**: Rejected because Flutter's rendering engine (Skia) provides more consistent cross-platform UI, the Dart language offers better null safety and async/await patterns than JavaScript, and Flutter's offline storage ecosystem (Hive, Isar) is more mature than React Native's options. The team also evaluated developer experience and found Flutter's hot reload significantly faster.
- **Native Android + iOS (Kotlin/Swift)**: Rejected due to the cost of maintaining two codebases with a small team. While native development provides the best platform-specific experience, iFarm's UI requirements do not justify the 2x development cost.
- **PWA (Progressive Web App)**: Rejected because PWAs on Android have limited access to device features (camera for photo capture, GPS for farm location), unreliable offline support across browser versions, and cannot be easily distributed via app stores, which is the expected installation path for iFarm's target users.
- **Kotlin Multiplatform (KMP)**: Rejected because KMP's UI sharing story (Compose Multiplatform) was not yet stable at the time of the decision, and the team has no Kotlin expertise.
- **Ionic/Capacitor**: Rejected due to performance concerns for offline-heavy data synchronization and inferior scrolling/animation performance compared to Flutter's compiled rendering.

## Consequences

- A single codebase serves both Android and iOS, reducing development and maintenance effort by an estimated 40%.
- Offline-first architecture ensures farmers can use the app in areas with poor connectivity, which is critical for rural Brazilian users.
- The team must learn Dart, which is a new language for the backend-focused TypeScript developers, though Dart's syntax is similar enough to reduce the learning curve.
- Shared TypeScript types from the monorepo (see ADR-008) cannot be directly used in Dart; API contracts must be maintained via OpenAPI specs or code generation.
- Flutter's release cycle is rapid; the team must allocate time for framework upgrades to avoid falling behind on security patches.
- App size is larger than native apps (~15-20MB) but within acceptable limits for the target audience.
